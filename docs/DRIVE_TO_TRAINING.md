# Google Drive → Training Pipeline

End-to-end recipe for turning a Google Drive corpus of SOPs and scanned files into a fine-tuned chat model for the platform.

## What this assumes

- A construction-project Drive folder structured like `100-Master / 200-Project Controls / 300-Delivery Management / ... ` with one to several levels of subfolder nesting
- A service account with read access to that folder (Drive shared with the service account's email)
- A GPU host to run the eventual fine-tune (PR 3a — see `docs/EDGE_PORT.md` for the deferred Orin port)

## Step 1: Configure Drive ingest

Two environment variables. The Fork already supports both; PR 25 added the recursive walker that makes deep folder trees actually work.

```bash
# Service-account JSON, either inline or a path
export GDRIVE_SERVICE_ACCOUNT_JSON='{"type":"service_account",...}'
# or:
export GDRIVE_SERVICE_ACCOUNT_JSON='/etc/secrets/sa.json'

# Map your platform project_id to the Drive root folder id
export GDRIVE_PROJECT_FOLDERS='construction_proj=1AbC...XyZ,other_proj=1DeF...UvW'
```

Find the folder ID from the Drive URL: `https://drive.google.com/drive/folders/1AbC...XyZ` → the ID is `1AbC...XyZ`.

## Step 2: Index the corpus

Trigger hydration manually (or let the nightly scheduler do it):

```bash
curl -X POST http://localhost:8000/v1/execute \
  -H "Authorization: Bearer cb_dev_key" \
  -d '{"block":"learning_engine","input_data":{"operation":"hydrate"}}'
```

The recursive walker pulls every leaf file (PDFs, DOCX, scans) from the Drive tree, the doc_index extracts text (with Tesseract OCR for scans), and chunks are persisted. Re-runs are idempotent — only new files are processed.

Check progress:

```bash
curl http://localhost:8000/v1/execute \
  -H "Authorization: Bearer cb_dev_key" \
  -d '{"block":"learning_engine","input_data":{"operation":"hydration_latest","scope":"project","project_id":"construction_proj"}}'
```

## Step 3: Generate training scenarios

Turn the indexed chunks into (question, expert-answer) JSONL pairs ready for fine-tuning:

```bash
python scripts/generate_training_scenarios.py \
  --project-id construction_proj \
  --out data/learning/training_scenarios.jsonl \
  --questions-per-chunk 3 \
  --min-chunk-chars 200
```

Each chunk gets 3 synthetic Q&A pairs generated by the chat block (DeepSeek by default; configurable via `--provider`). The script:

- Skips chunks shorter than `--min-chunk-chars` (avoids garbage from OCR artifacts on cover pages)
- Refuses to emit pairs when the chat block falls back to the offline template (no LLM = no synthetic data, rather than nonsense)
- Filters out trivially-short instructions or responses (LLM partial outputs)
- Strips code-fence wrappers from JSONL responses

Expected output: ~3× the chunk count, minus filter losses. A medium corpus (1000 chunks) produces ~2500 pairs in 5-15 minutes depending on provider rate limits.

## Step 4: Fine-tune

```bash
pip install -r requirements-ml.txt  # if not already
python scripts/finetune_router.py \
  --train-data data/learning/training_scenarios.jsonl \
  --epochs 3 \
  --lora-r 16
```

Saves a LoRA adapter to `data/learning/adapters/construction_v1/`. PR 3a's `local_model.py` picks it up automatically.

## Step 5: Use the fine-tuned model

```bash
curl -X POST http://localhost:8000/v1/execute \
  -H "Authorization: Bearer cb_dev_key" \
  -d '{"block":"chat","input_data":{"text":"what is the minimum concrete cover for a slab in moderate exposure?","use_local_model":true}}'
```

Response should carry `"provider": "local_lora"` and use vocabulary specific to your SOPs (e.g. referencing the exact codes and standards your documents cite).

## Quality controls

The synthetic Q&A is only as good as the LLM that generated it. To sanity-check before paying for a full fine-tune:

1. After step 3, open `data/learning/training_scenarios.jsonl` and read 20 random rows. If the answers are wrong or generic, the chosen provider isn't strong enough — try a better LLM or upgrade the prompt template (see `_DEFAULT_PROMPT` in `scripts/generate_training_scenarios.py`).
2. Reserve a small held-out set BEFORE the fine-tune (e.g. `head -50` of the JSONL into a separate file, then strip those from the training set). Compare local vs cloud answers on those 50 questions after the fine-tune — that's your accuracy proxy.

## What this does NOT cover

- **Encrypted documents on Drive**: the service account needs read access, not write — and any client-side encryption (e.g. Drive's confidential mode) will fail to download. Handle on the Drive side first.
- **Google-native Docs/Sheets/Slides**: the current Drive ingest skips these (they need an `export` call instead of `alt=media`). Export them to PDF first or extend `gdrive_service.download_file_bytes` to handle the export path. Documented as a known gap.
- **Real-time updates**: hydration is a nightly cron. For "I added a file an hour ago and want it indexed now", call `hydrate` manually as in step 2.
